# psms-service plan.md

## 문서 목적

이 문서는 PSMS(Private Stock Monitoring Service)의 실제 구현 계획과 기술 설계 방향을 정리하기 위한 문서이다.

현재 프로젝트는 **국내시장 정시 체크용 보조 웹 대시보드**를 만드는 것이 목적이며,  
HTS 전체를 대체하는 것이 아니라 **핵심 표를 웹에서 빠르게 확인할 수 있게 재구성하는 것**에 집중한다.

초기 범위에는 아래 두 축이 함께 포함된다.

- 웹 대시보드
- 텔레그램 링크 리마인더

즉, 웹이 주 서비스인 것은 맞지만  
텔레그램 리마인더도 **초기부터 함께 포함하는 기능**으로 본다.

---

## 문서 역할

이 문서는 PSMS의 설계 방향, 확정 사항, 그리고 이후 사용자가 결정해야 하는 설계 사안을 정리한다.

정리 대상은 다음과 같다.

- 서비스 범위
- 시스템 아키텍처 방향
- 데이터 수집 및 저장 전략
- 화면 구성 방향
- 백엔드 / 프론트엔드 역할 분리
- 텔레그램 기능 범위
- 보안 / 환경설정 원칙
- 구현 단계와 우선순위
- 앞으로 추가 결정이 필요한 설계 사안

반대로, 리눅스 서버에서 직접 수행해야 하는 설치/설정 작업은 `setup-todo.md`에서 관리한다.

예를 들면 아래는 `setup-todo.md`의 관리 대상이다.

- 서버에서 직접 해야 하는 Docker / Compose 적용
- Nginx 설정 파일 반영
- 운영 DB 생성 및 계정 권한 처리
- 운영 환경변수 반영
- 실제 배포 서버 점검 작업

즉, 이 문서는 **무엇을 어떻게 만들지 정하는 설계 문서**이고,  
`setup-todo.md`는 **사용자가 서버에서 직접 수행해야 하는 작업 문서**다.

## 문서 구분 원칙

문서가 길어지더라도 아래 기준은 유지한다.

### `plan.md`에 남겨야 하는 것
- 이미 합의된 설계 방향
- 왜 그런 방향을 택했는지에 대한 이유
- 지금은 하지 않기로 한 항목과 그 이유
- 추후 사용자가 결정해야 하는 설계 사안
- 다음 세션이나 구현자가 그대로 따라갈 수 있도록 필요한 상세 설명

### `setup-todo.md`로 보내야 하는 것
- 실제 서버에서 직접 해야 하는 작업
- 배포 환경에서 수동으로 처리해야 하는 작업
- 자동화 대상이 아니고 사용자가 인지하고 실행해야 하는 작업

즉, `plan.md`는 요약본이 아니라 **설계 기준서**로 유지한다.

## 1. 프로젝트 목표

### 1-1. 만들고자 하는 것
- 국내시장 핵심 표를 정시 기준으로 빠르게 확인할 수 있는 웹 대시보드
- HTS 여러 화면을 일일이 열지 않고도 핵심 시장 흐름을 볼 수 있는 보조 도구
- 정해진 시각에 대시보드 확인을 유도하는 텔레그램 링크 리마인더

### 1-2. 만들지 않는 것
- HTS 전체 기능 복제
- 주문 / 계좌 / 매매 기능
- 초단위 트레이딩 도구
- 범용 차트 플랫폼
- 장문의 텔레그램 요약 메시지 서비스

### 1-3. 핵심 원칙
- HTS UI 복제가 아니라 **핵심 정보 재구성**이 목적이다.
- 화면 요청 시 외부 API를 직접 호출하지 않는다.
- 외부 데이터는 사전에 수집 / 가공 / 저장하고, 웹은 저장된 데이터를 조회한다.
- 텔레그램은 요약 텍스트 발송보다 **대시보드 진입 링크 리마인더**에 우선 집중한다.
- 초기 데이터 소스는 **키움 REST API**를 우선 사용한다.
- 구현은 나중으로 미루더라도, 설계와 저장 구조는 먼저 최대한 분명하게 정리한다.

## 2. 프로젝트 분리 원칙

### 2-1. PSMS와 PTAS의 관계
PSMS와 기존 PTAS는 현재 단계에서 논리적으로 분리해서 진행한다.

즉,
- 같은 서버를 쓸 수는 있다.
- 일부 공통 인프라를 나중에 공유할 수는 있다.
- 그러나 현재 설계 기준에서는 **별도 프로젝트**로 본다.

### 2-2. 텔레그램 기능의 위치
텔레그램 기능은 PSMS 바깥의 별도 보조 프로젝트가 아니라,  
**PSMS 초기 범위 안에 포함되는 기능**으로 본다.

다만 텔레그램이 메인이 아니라, 어디까지나 아래 역할로 제한한다.

- 사용자가 정해진 시각에 체크를 잊지 않도록 알림
- 링크를 통해 웹 대시보드로 진입 유도
- 필요한 경우 기준 시각이나 짧은 한 줄 상태 전달

즉,  
**PSMS = 웹 대시보드 + 텔레그램 링크 리마인더를 함께 포함하는 서비스**  
라는 전제로 본다.

## 3. 시스템 구조 방향

## 3-1. 전체 구성

Browser  
↓  
Nginx  
↓  
React 정적 리소스 제공 / Spring Boot API reverse proxy  
↓  
Spring Boot Backend  
↓  
MariaDB

### 3-2. 구조를 이렇게 잡는 이유
이 구조는 학습 목적과 실제 운영 편의 둘 다를 고려한 선택이다.

- React + Spring Boot 기반 웹 서비스 경험
- Reverse Proxy 구조 경험
- Docker 기반 운영 구조 경험
- 프론트 / 백엔드 / DB 역할 분리 경험

또한 운영 단계에서 아래 장점이 있다.

- 정적 파일 서빙과 API 진입점을 분리할 수 있음
- Nginx를 통한 라우팅 구조를 명확히 가질 수 있음
- 추후 HTTPS 적용, 캐싱, 압축 같은 확장 포인트가 생김

### 3-3. 컨테이너 구성
- `nginx`
- `backend`
- `mariadb`

비고
- React는 별도 장기 실행 컨테이너보다, 빌드 산출물을 Nginx가 서빙하는 방향을 우선 검토한다.
- 추후 개발 편의를 위해 dev 환경에서만 프론트 개발 서버를 별도로 둘 수 있다.

### 3-4. 배포 방향
- 운영 서버(1GB)는 이미지 실행만 담당한다.
- 운영 서버에서 매번 직접 빌드하지 않는다.
- 빌드는 외부 환경(GitHub Actions 또는 별도 빌드 서버)에서 수행하고, 운영은 pull 후 재기동하는 방향을 우선 검토한다.

이 방향은 운영 서버 자원이 크지 않다는 점과,  
운영 환경에서는 “빌드”보다 “기동”에 집중하는 편이 더 안정적이라는 판단 때문이다.

## 5. 화면 구성 방향

## 5-1. 메인 대시보드
메인 대시보드는 사용자가 정시 체크 시 가장 먼저 들어오는 화면이다.

구성 방향
- 상단: 기준 시각, 마지막 수집 시각, 시장 상태 요약
- 1영역: 시장종합
- 2영역: 투자자별 매매종합
- 3영역: 장중 투자자별 매매 상위
- 4영역: 관심 종목 중심의 프로그램매매추이 진입
- 5영역: 관심 종목 중심의 공매도추이 진입

설계 원칙
- 한 화면에 너무 많은 HTS 요소를 그대로 옮기지 않는다.
- 핵심 카드 + 표 조합으로 단순화한다.
- 정시 확인 순서에 맞는 섹션형 구성을 우선한다.
- “많이 보여주는 것”보다 “빨리 읽히는 것”을 우선한다.

### 5-2. 메인 화면에서 보여줘야 하는 핵심 정보
사용자가 첫 화면에서 최소한 아래는 바로 이해할 수 있어야 한다.

- 지금 보고 있는 데이터의 기준 시각이 언제인지
- 실제 마지막 수집 성공 시각이 언제인지
- 시장 전체가 강한지 약한지
- 투자 주체별로 누가 사고 누가 파는지
- 장중 강하게 움직이는 종목이 무엇인지
- 관심 종목 중 프로그램 / 공매도 관점에서 눈에 띄는 종목이 무엇인지

즉, 첫 화면은 “시황 페이지”라기보다  
**정시 점검을 위한 요약 상황판**에 가깝다.

## 5-3. 상세 화면
각 표는 별도 상세 화면 또는 섹션 확장 형태로 제공한다.

예상 상세 화면
- 시장종합 상세
- 투자자별 매매종합 상세
- 장중 투자자별 매매 상위 상세
- 종목별 프로그램매매추이 상세
- 종목별 공매도추이 상세

상세 화면에서는 아래를 고려한다.

- 정렬
- 필터
- 관심 종목 중심 조회
- 시점 선택 또는 최근 구간 조회
- 표 중심 레이아웃 유지

## 5-4. 텔레그램 리마인더 연결
텔레그램 메시지는 아래 역할을 한다.

- 지정 시각에 알림 발송
- 현재 시점 대시보드로 이동할 수 있는 링크 제공
- 필요 시 기준 시각 또는 간단한 한 줄 상태만 포함

초기에는 아래를 제외한다.
- 상위 N개 종목을 텍스트로 길게 발송
- 표 내용을 메시지 본문에 대량 포함
- 메시지 자체를 소형 대시보드처럼 확장

## 6. 데이터 소스 및 수집 방향

## 6-1. 기본 원칙
- 1차 데이터 소스는 **키움 REST API**
- KRX는 필요 시 보조 소스 성격으로만 검토
- 초기에는 여러 외부 API를 복잡하게 조합하지 않는다.

### 6-2. 키움을 1차로 두는 이유
현재 과제의 핵심은 **국내시장 HTS 표 재구성**이다.

이 관점에서 키움은
- 지금 만들려는 표 구조와 더 가깝고
- 실제 사용 흐름과도 잘 맞으며
- 사용자가 계속 사용할 가능성이 높은 증권사 기반이라는 장점이 있다.

즉,  
“범용성 최우선”보다 “현재 만들려는 기능과의 적합성”을 우선한 선택이다.

## 6-3. 키움 API 매핑 방향

### 비교적 직접 매핑 가능한 영역
- `stock_master`
  - `ka10095`
  - `ka10099`
  - `ka10100`
- 장중 투자자별 매매 상위
  - `ka10062`
  - `ka10098`
- 종목별 프로그램매매추이
  - `ka90007`
  - `ka90010`
  - 실시간 `0u`
- 공매도추이
  - `kt50075`

### 서버에서 내부 표준 모델로 재구성할 가능성이 있는 영역
-시장종합
-투자자별 매매종합

즉 초기 구현은 아래 방식으로 본다.
- 직접 매핑 가능한 표는 API 응답을 내부 모델로 정규화해서 저장
- 직접 화면 단위가 없는 표는 여러 응답을 모아 서버에서 표 형태로 가공
- HTS와 수치/컬럼이 완전히 동일하지 않더라도, 실사용에 필요한 구조를 우선한다.

## 6-4. 서버 가공이 필요한 표의 의미
시장종합과 투자자별 매매종합은  
단일 API 응답 하나를 그대로 보여주는 방식보다, 여러 데이터 조각을 묶어 **웹에서 보기 좋은 표 구조**로 재구성할 필요가 있다.

이때 중요한 원칙은 다음과 같다.

- API 중심이 아니라 화면 중심으로 내부 모델을 설계한다.
- 외부 응답 포맷을 그대로 프론트에 노출하지 않는다.
- 컬럼명 / 단위 / 표현 형태는 내부 표준 기준으로 맞춘다.
- HTS와 완전히 똑같이 만들기보다, 실제 사용자가 빨리 판단할 수 있게 만드는 것을 우선한다.

## 7. 저장 구조 방향

## 7-1. 현재 합의된 테이블 방향
- `stock_master`

- `market_overview`
- `market_overview_snapshot`

- `investor_trading_summary`
- `investor_trading_summary_snapshot`

- `intraday_investor_ranking_snapshot`

- `program_trading_history`
- `short_selling_history`

## 7-2. 테이블 역할

### `stock_master`
- 종목 코드, 종목명, 시장 구분 등 종목 기준 정보 저장
- 키움 API 기준 daily update 수행

### `market_overview`
- 가장 최신 기준의 시장종합 요약 데이터
- 대시보드 첫 진입 시 빠른 조회용

### `market_overview_snapshot`
- 기준 시각별 시장종합 이력 저장
- 특정 시점 대시보드 재구성 시 사용

### `investor_trading_summary`
- 가장 최신 기준의 투자자별 매매종합 데이터
- 빠른 조회용 latest 테이블

### `investor_trading_summary_snapshot`
- 기준 시각별 투자자별 매매종합 이력 저장

### `intraday_investor_ranking_snapshot`
- 특정 시각 기준 투자자 주체별 상위 종목 랭킹 저장
- 성격상 latest보다 snapshot 중심이 더 적절

### `program_trading_history`
- 종목별 / 시각별 프로그램매매 추이 저장
- 상세 조회 시 시계열 데이터로 사용

### `short_selling_history`
- 종목별 / 일자 기준 공매도 추이 저장
- 최근 N일 표 중심 조회에 사용

## 7-3. latest / snapshot / history를 나누는 기준
초기 설계에서 중요한 것은 모든 데이터를 똑같이 저장하지 않는 것이다.

### latest 성격이 필요한 경우
- 대시보드 첫 화면에서 빠르게 바로 보여줘야 하는 표
- 항상 “현재 기준 최신값”이 중요한 표
- 예: `market_overview`, `investor_trading_summary`

### snapshot 성격이 필요한 경우
- 특정 기준 시각 단위로 상태를 다시 보고 싶은 표
- 정시 점검 기록을 남겨야 하는 표
- 예: `market_overview_snapshot`, `investor_trading_summary_snapshot`, `intraday_investor_ranking_snapshot`

### history 성격이 필요한 경우
- 시계열 흐름 자체가 본질인 표
- 최근 N개 구간이나 기간 추이를 봐야 하는 표
- 예: `program_trading_history`, `short_selling_history`

이 구분을 명확히 해두면,
- 저장 정책
- 조회 API 구조
- 인덱스 방향
- 프론트 화면 사용 방식

이 모두가 자연스럽게 정리된다.

## 7-4. 공통 시간 컬럼 규칙
- `snapshot_time`
  - 논리적 기준 시각
  - 예: 09:59:30에 수집이 끝났더라도 10시 점검용이면 `10:00:00`
- `created_at`
  - 실제 저장 시각

이 구분은 나중에 매우 중요하다.

사용자가 보고 싶은 것은 “실제 DB 저장 순간”보다  
**어느 기준 시각의 데이터인지**이기 때문이다.

## 7-5. enum / 메타 관리 원칙

### enum으로 관리
- `InvestorType`
- `MarketType`

### 별도 테이블로 두지 않는 것
- 투자자 구분 코드성 테이블
- 시장 구분 코드성 테이블

초기에는 코드성 기준 정보까지 전부 테이블화하지 않는다.  
변경 빈도가 낮고, 애플리케이션 레벨에서 관리하는 편이 더 단순하기 때문이다.

## 7-6. 지금 하지 않는 것
- Redis 도입
- collection / snapshot header 전용 테이블 추가

비고
- 복잡도 증가를 피하기 위해 초기에는 제외한다.
- 추후 snapshot 묶음 관리가 실제로 필요해질 때 재검토한다.

### Redis를 지금 쓰지 않는 이유
현재 프로젝트는 실시간 초저지연 처리보다,  
정시 기준으로 수집한 데이터를 안정적으로 보여주는 쪽이 더 중요하다.

따라서 초기에는
- 저장 단순성
- 구조 명확성
- 운영 복잡도 최소화

를 우선하고, latest 데이터도 MariaDB로 관리한다.

### snapshot header를 두지 않는 이유
여러 번 논의했지만, 현재는 `snapshot_time`을 논리적 기준 시각으로 공통 저장하면 초기에는 충분하다고 본다.

즉,
- 별도 스냅샷 그룹 엔터티를 만들지 않아도 되고
- 각 표가 동일 기준 시각으로 저장되면
- 조회 시점에서도 충분히 묶어 해석할 수 있다

다만 추후 아래 상황이 생기면 재검토할 수 있다.

- 여러 표를 한 번의 수집 배치 묶음 단위로 명확히 관리해야 할 때
- 부분 성공 / 부분 실패를 별도 그룹 단위로 추적해야 할 때
- 특정 수집 배치 ID 기준 운영 점검이 꼭 필요해질 때

## 8. 수집 / 스케줄링 설계 방향

## 8-1. 수집 작업 종류
1. 종목 마스터 동기화
2. 시장종합 수집
3. 투자자별 매매종합 수집
4. 장중 투자자별 매매 상위 수집
5. 종목별 프로그램매매추이 수집
6. 종목별 공매도 데이터 수집
7. 텔레그램 리마인더 발송

## 8-2. 스케줄러 위치
- 초기에는 별도 배치 서버 없이 `backend` 내부 스케줄러로 처리
- 수집과 조회를 분리
- 웹 요청 시 외부 API 직접 호출 금지

### 이렇게 가는 이유
초기 단계에서 중요한 것은 시스템을 과도하게 쪼개지 않고,  
먼저 동작 흐름을 명확히 만드는 것이다.

따라서 지금은
- collector와 API 서버를 분리한 별도 프로세스
- 별도 배치 전용 인스턴스
- 메시지 큐 기반 복잡한 수집 구조

까지는 가지 않는다.

필요해지면 나중에 분리할 수 있도록 설계만 열어두고,  
초기에는 `backend` 내부에서 처리한다.

## 8-3. 수집 주기 방향
정확한 주기는 추후 확정하지만 현재 기준 방향은 아래와 같다.

- 시장 표 수집: 시장 시간대에 5분 ~ 10분 단위 우선 검토
- 종목 마스터: 일 1회
- 텔레그램 리마인더: 정시 또는 사용자가 정한 시각 기준 발송

## 8-4. 수집 기준 시각 개념
이 프로젝트에서 중요한 것은 “실시간 틱 처리”가 아니라  
**정해진 기준 시각에 맞춘 점검 데이터 확보**다.

예를 들어 10시 점검 데이터를 본다면,
- 실제 수집 완료가 09:59:40일 수도 있고
- 10:00:15일 수도 있다.

하지만 논리적으로는 이것을 “10시 기준 데이터”로 묶어야 한다.

즉, 스케줄링은 단순 cron 문제가 아니라  
**점검 기준 시각과 저장 기준 시각을 어떻게 맞출지**의 문제다.

## 8-5. 저장 원칙
- 수집 성공 시 내부 공통 모델로 정규화
- latest가 필요한 표는 latest 테이블 갱신
- 시점 조회가 필요한 표는 snapshot/history도 함께 저장
- 부분 실패 시 재시도 및 실패 로그 정책은 추후 세부화

## 8-6. 앞으로 더 결정할 수집 관련 사안
- 정시 기준 시각을 몇 시 몇 분 단위로 운영할지
- 장 시작 전 / 장중 / 장 마감 후 수집 구간을 어떻게 나눌지
- 공매도 데이터가 일 단위인지 장중 단위인지에 따라 저장 기준을 어떻게 나눌지
- 실패 허용 범위와 재수집 정책을 어느 정도까지 둘지

## 9. 백엔드 설계 방향

## 9-1. 백엔드 역할
- 키움 API 연동
- 외부 응답 정규화 및 가공
- 내부 저장
- 대시보드 조회 API 제공
- 텔레그램 링크 리마인더 발송
- 운영용 스케줄링 수행

## 9-2. API 역할 구분
초기 API는 아래 성격으로 분리한다.

### 조회 API
- 대시보드 메인 조회
- 표별 상세 조회
- 특정 종목 프로그램매매 추이 조회
- 특정 종목 공매도 추이 조회
- 특정 시점 기준 조회

### 내부/운영 API 또는 관리 기능
- 수집 상태 확인
- 마지막 성공 수집 시각 확인
- 수동 재수집 트리거 여부는 추후 검토

## 9-3. 프론트 친화적 응답 원칙
API는 DB 구조를 그대로 노출하는 방식보다,  
프론트가 바로 렌더링하기 쉬운 형태를 우선한다.

즉,
- 화면 단위 응답 구성
- 기준 시각 / 마지막 수집 시각 / 상태를 함께 전달
- 표 렌더링에 필요한 컬럼 중심 구조
- 내부 enum / raw 코드 노출 최소화

이 원칙을 지키면 프론트가 단순해지고,  
DB 구조 변경이 API에 바로 전파되는 문제도 줄일 수 있다.

## 9-4. 백엔드 내부 처리 계층 방향
구현 전에 세부 패키지 구조를 더 정리하겠지만, 큰 역할은 아래처럼 분리한다.

- external
  - 키움 API 연동
- collector
  - 데이터 수집 orchestration
- domain
  - 표별 비즈니스 모델 / 저장 / 조회
- notification
  - 텔레그램 리마인더
- api
  - 외부 조회 API
- support
  - 공통 유틸 / 시간 / 예외 / 설정

비고
- 현재 단계에서는 코드 작성보다 설계와 문서화를 먼저 끝내는 것이 우선이다.

## 9-5. 도메인 분리 원칙
표마다 성격이 다르므로, 한 서비스에 모든 수집/조회 로직을 몰아넣지 않는다.

예를 들면 아래처럼 보는 것이 자연스럽다.

- 시장종합 도메인
- 투자자별 매매종합 도메인
- 장중 투자자 랭킹 도메인
- 프로그램매매 도메인
- 공매도 도메인
- 종목 마스터 도메인

즉,  
“표 단위 기능”을 도메인 경계로 보고 구조를 나누는 편이 이후 유지보수에 유리하다.

## 11. 텔레그램 기능 설계 방향

## 11-1. 초기 포함 범위
- 링크 리마인더 발송
- 지정 시각 알림 발송
- 대시보드 링크 포함
- 필요 시 기준 시각 포함

## 11-2. 발송 메시지 방향
예시 성격
- “10시 체크 시간입니다”
- “PSMS 대시보드 확인: [링크]”

즉, 메시지는 진입 유도에 집중한다.

## 11-3. 텔레그램을 이렇게 제한하는 이유
초기부터 텔레그램 본문에 많은 수치를 담기 시작하면 아래 문제가 생긴다.

- 메시지 포맷 설계가 커짐
- 표별 요약 규칙이 복잡해짐
- 링크보다 메시지 자체가 주 서비스처럼 변질될 수 있음
- 화면 / 데이터 구조보다 메시지 조립 로직이 더 커질 수 있음

PSMS의 중심은 웹 대시보드이므로,  
텔레그램은 그 진입을 돕는 보조 채널로 제한하는 것이 맞다.

## 11-4. 추후 확장 가능 항목
- 사용자별 구독 표 종류
- 특정 조건 충족 시 이벤트성 알림
- 텔레그램 내 간단 요약 블록

다만 이는 모두 2차 이후 검토 대상으로 본다.

- 구현보다 설계 / 문서화 / 구조 확정을 먼저 끝내는 흐름으로 진행한다.

## 기술 스택 및 구현 기준 확정

### 1. 백엔드 기술 스택
본 프로젝트의 백엔드는 Spring Boot 기반으로 구현한다.

- Java 버전은 21을 사용한다.
- 빌드 도구는 Gradle을 사용한다.
- Gradle DSL은 Groovy DSL 기준으로 구성한다.
- 백엔드 데이터 접근 계층은 JPA를 기본으로 사용한다.
- 단순 CRUD 및 일반적인 조회는 JPA 기준으로 처리한다.
- 단, 조건 조합이 복잡하거나 동적 조회가 필요하여 JPA만으로 가독성 또는 유지보수성이 떨어지는 경우 Querydsl을 함께 사용한다.

즉, 기본 방향은 "JPA 우선, 필요 시 Querydsl 보완"이다.  
처음부터 모든 조회를 Querydsl로 강제하지는 않으며, 실제 구현 과정에서 필요한 영역에 한해 점진적으로 적용한다.

### 2. 프론트엔드 기술 스택
프론트엔드는 React 기반으로 구성한다.

- React 사용
- Vite 사용
- TypeScript 사용

초기 단계에서는 프론트엔드와 백엔드를 함께 관리하는 단일 레포 기준으로 진행한다.  
현재 프로젝트의 핵심은 프론트엔드 기술 실험이 아니라, HTS 표 형태의 정보를 웹에서 확인할 수 있는 개인용 대시보드 구현이므로, 프론트엔드 구조 또한 과도한 추상화보다는 빠르게 화면을 구성하고 수정하기 쉬운 방향을 우선한다.

### 3. DB 및 마이그레이션 관리
DB 스키마는 수동 반영 방식이 아니라 마이그레이션 도구를 통해 관리한다.

- 마이그레이션 도구는 Flyway 사용
- DDL은 버전 기반 migration 스크립트로 관리
- 운영/개발 환경 모두 동일한 migration 체계를 따른다

즉, 테이블 생성/변경/인덱스 추가 등의 스키마 변경 사항은 SQL 파일 버전 관리 방식으로 누적하며, 수동으로 운영 DB에 직접 반영하는 방식은 기본 운영 방식으로 채택하지 않는다.

이렇게 하는 이유는 다음과 같다.

- latest / snapshot / history 성격의 테이블이 단계적으로 늘어날 가능성이 높다.
- 프로토타입 단계에서도 스키마 변경 이력을 추적할 수 있어야 한다.
- 향후 운영 서버 재구성, 신규 환경 구성, 테스트 환경 분리 시 동일한 구조를 재현하기 쉬워야 한다.

## 배포 및 운영 방식 확정

### 1. 배포 구조
운영 서버는 빌드 전용 서버가 아니라 실행 전용 서버로 본다.

배포 흐름은 아래 기준으로 정리한다.

1. 외부 빌드 환경에서 애플리케이션을 빌드한다.
2. Docker 이미지를 생성한다.
3. 생성한 이미지를 이미지 레지스트리에 push 한다.
4. 운영 서버는 해당 이미지를 pull 한다.
5. 운영 서버에서 컨테이너를 재생성하여 기동한다.

초기 배포 방식은 복잡한 무중단 전환 구조가 아니라, 단순한 pull + recreate 방식으로 진행한다.

여기서 말하는 recreate는 다음 의미를 가진다.

- 기존 컨테이너를 유지한 채 트래픽만 옆으로 넘기는 blue-green 방식이 아님
- 새 이미지를 pull 받은 뒤, 해당 서비스를 새 이미지 기준으로 다시 생성하여 기동하는 방식
- 즉, 현재 프로젝트에서는 배포 구조 단순성과 운영 편의성을 우선한다

본 프로젝트는 초기 기준으로 소규모 개인용 서비스이며, 운영 규모 또한 매우 작기 때문에 배포 복잡도를 과도하게 높이지 않는다.

### 2. 이미지 레지스트리 및 시크릿 관리
- 이미지 레지스트리는 GHCR 사용
- 민감 정보는 GitHub Secrets 및 서버 환경 변수 파일로 관리
- 운영/개발 환경값은 분리 관리

민감 정보 예시는 다음과 같다.

- DB 접속 정보
- 키움 API 관련 인증 정보
- 텔레그램 봇 토큰 및 채팅방 정보
- 대시보드 base URL
- 기타 외부 연동용 비밀값

민감 정보는 소스코드나 일반 설정 파일에 하드코딩하지 않으며, 환경별 주입 방식으로 관리한다.

### 3. 접근 제어
초기 대시보드는 정식 사용자 인증/인가 시스템을 도입하지 않는다.  
다만 아무 제약 없이 외부에 완전히 노출하는 구조도 지양한다.

초기 접근 제어는 아래와 같이 정리한다.

- Nginx Basic Auth 수준의 간단 보호 적용
- 회원가입/로그인/권한 체계는 초기 범위에 포함하지 않음
- 본 프로젝트는 개인용 프로토타입 성격이므로, 우선은 최소한의 보호 장치만 적용

즉, 현재 단계의 목표는 정식 서비스 수준의 인증 체계 구축이 아니라, 개인 운영 관점에서 무방비 공개를 피하는 정도의 최소 보호이다.

## 기능 범위 및 단계 구분 확정

### 1. 대상 표 범위
본 프로젝트의 대상 표는 총 7개이며, 모두 구현 범위에 포함한다.

즉, 일부 표는 제외 대상이 아니며, 전체 7개를 최종 구현 대상으로 본다.

### 2. 1차 / 2차 구분의 의미
기존에 일부 항목을 1차 MVP, 일부 항목을 2차 검토로 나누었던 것은 구현 제외 여부 때문이 아니라, 데이터 처리 난이도 차이를 구분하기 위함이다.

정확한 의미는 다음과 같다.

- 1차 영역: 외부 API 응답값을 상대적으로 직접적으로 가공하여 표 구성 가능
- 2차 영역: 서버 내부에서 추가 계산, 집계, 정렬, 조합 로직을 수행해야 최종 표를 만들 수 있음

따라서 1차/2차 구분은 "한다 / 안 한다"의 구분이 아니라, "상대적으로 직접 구성 가능 / 서버 연산이 더 필요함"의 구분으로 이해한다.

즉, 구현 범위는 7개 전체이며, 단계 구분은 내부 처리 복잡도 구분이다.

### 3. 서버 가공이 필요한 표에 대한 원칙
서버 가공이 필요한 표는 단순 원본 중계가 아니라, 서버에서 다음 중 하나 이상의 작업이 필요할 수 있다.

- 정렬 기준 재구성
- 지표 계산
- 파생 컬럼 생성
- 랭킹 계산
- 여러 응답값 조합
- 화면 표시용 결과셋 재구성

이 경우에도 화면 요구사항에 맞는 최종 결과를 서버에서 책임지고 제공하는 방향으로 설계한다.  
즉, 프론트엔드에서 과도한 계산을 수행하기보다, 백엔드가 화면에 필요한 형태에 가깝게 가공한 데이터를 제공하는 방향을 우선한다.

## 프로토타입 원칙 및 사용자 범위

### 1. 프로토타입 원칙
현재 단계는 프로토타입 단계로 본다.

이는 아래 의미를 포함한다.

- 실제 사용 과정에서 요구사항이 충분히 바뀔 수 있음
- 수집 주기, 저장 범위, 화면 구성, 컬럼 구조, 표시 방식이 조정될 수 있음
- 초기 설계는 완전 고정보다 수정 용이성을 우선함
- 다만 프로토타입이라는 이유로 구조를 무질서하게 만들지는 않음

즉, 지금 단계의 핵심은 "미래의 모든 경우를 미리 커버하는 구조"가 아니라, "현재 요구사항을 기준으로 충분히 명확하고 수정 가능한 구조"를 만드는 것이다.

### 2. 사용자 범위에 대한 기준
현재 서비스는 기본적으로 1인용 개인 대시보드 성격으로 본다.  
즉, 현재 기획 기준은 불특정 다수가 사용하는 공개형 서비스가 아니라, 개인이 직접 사용하는 웹 대시보드이다.

다만 아래 기준은 함께 유지한다.

- 완전히 단일 사용자 하드코딩 구조로 만들지는 않음
- 사용자 구분 가능성 정도는 최소한 열어둠
- 향후 2~3명 수준의 소규모 확장 가능성을 완전히 막지 않음

즉, 기획은 1인용 기준으로 단순하게 가져가되, 데이터 구조는 최소한의 사용자 식별 기반을 둘 수 있게 한다.

이 원칙은 아래 영역에 적용한다.

- 사용자 설정
- 관심종목
- 알림 설정
- 리마인더 시각
- 향후 필요한 개인화 설정 항목

## 사용자 설정 및 리마인더 정책 확정

### 1. 리마인더 기능 방향
텔레그램은 대시보드 링크 리마인더 역할로 사용한다.

즉, 텔레그램의 주 목적은 자체적인 상세 데이터 소비가 아니라, 사용자가 적절한 시점에 대시보드에 진입하도록 유도하는 것이다.

초기 기준 리마인더는 고정 시각이 아니라 사용자 설정값 기반으로 관리한다.

### 2. 리마인더 시각 저장 방식
리마인더 시각은 코드 상수나 설정 파일 고정값이 아니라 DB 테이블에 저장한다.

초기에는 사용자별 알림 설정 테이블을 두는 방향으로 설계한다.  
예를 들어 아래 성격의 정보를 저장할 수 있다.

- 사용자 식별값
- 알림 사용 여부
- 리마인더 시각
- 타임존
- 생성일시
- 수정일시

초기에는 리마인더 시각 1개만 지원하는 단순 구조로 시작한다.  
즉, 사용자당 하나의 대표 리마인더 시각을 저장하는 방식으로 본다.

향후 필요 시 다음 확장을 검토할 수 있다.

- 여러 리마인더 시각 지원
- 요일별 리마인더
- 장중/장마감 구분 리마인더
- 그룹별 리마인더

하지만 현재 단계에서는 그런 확장 요구를 선반영하지 않고, 단일 시각 기반 구조로 시작한다.

### 3. 리마인더 발송 방식
리마인더 발송은 "사용자 설정 시각과 현재 시각을 비교하는 방식"으로 처리한다.

초기 구현 방향은 다음과 같다.

- 스케줄러는 매분 실행
- 현재 시각을 기준으로 알림 활성 사용자 설정 조회
- 사용자 설정 시각과 일치하는 경우 텔레그램 발송

즉, 동적으로 사용자별 스케줄을 등록/해제하는 복잡한 방식보다는, 매분 확인하는 단순한 방식으로 시작한다.

이 방식을 택하는 이유는 다음과 같다.

- 구현 단순성
- 사용자 설정 변경 반영이 쉬움
- 운영 중 재기동/설정 변경 시 복잡한 스케줄 재등록 로직이 필요 없음
- 현재 프로젝트 규모에서 성능 부담이 사실상 무시 가능함

### 4. 리마인더 기본 정책
초기 기본 정책은 아래와 같이 둔다.

- 리마인더 기준 시각은 매시 정각 개념을 기본값으로 본다
- 다만 실제 발송 시각은 사용자 설정값으로 관리한다
- 초기 기본 수집 주기는 5분으로 시작한다

즉, 시스템 기본 감각은 정시 기반 리마인더이지만, 실제 구현은 사용자별 설정값 저장 구조를 따른다.

## 관심종목 및 그룹 기능 정리

### 1. 관심종목 관리
관심종목은 초기부터 DB 테이블 기반으로 관리한다.

즉, 코드 내부 하드코딩 목록이나 설정 파일 상수로만 관리하지 않는다.  
이유는 다음과 같다.

- 종목 추가/삭제/변경 가능성이 높음
- 사용자 설정과 연결될 수 있음
- 향후 화면 기능과 직접 연결될 가능성이 큼

### 2. 그룹 기능 제외
과거 대화 흐름에서 관심종목 그룹, 그룹별 탭 전환 등의 가정성 기능이 있었으나, 이는 직접 요구사항이 확정되기 전 임시 추정에 기반한 내용이었다.

현재는 직접 요구사항을 기준으로 진행하며, 그룹 기능은 현재 범위에서 제외한다.

즉, 현재 plan 기준에서는 다음과 같이 정리한다.

- 관심종목은 관리 대상에 포함
- 그룹 기능은 현재 요구사항에 없으므로 제외
- 이후 실제 필요성이 생기면 별도 기능으로 추가 검토

## 데이터 저장 및 보존 정책 보완

### 1. 저장 전략 방향
데이터 저장은 단순 원본 누적이 아니라 표 성격에 따라 구분한다.

현재 기본 방향은 다음과 같다.

- latest 성격 데이터
- snapshot 성격 데이터
- history 성격 데이터

각 표는 화면 요구사항과 조회 패턴에 따라 적절한 저장 전략을 선택한다.

### 2. 보존 정책
초기 보존 정책은 프로토타입 기준으로 단순하게 시작한다.

우선 방향은 다음과 같이 둔다.

- snapshot 데이터는 30일 기준 보관
- history 데이터는 표 성격에 따라 이후 세분화 가능
- latest 데이터는 현재 상태 조회 중심으로 유지

즉, 초기에는 보존 정책까지 과도하게 복잡하게 만들지 않으며, 실제 사용량과 저장량을 보면서 후속 조정한다.

## 구현 시 주의할 공통 원칙

### 1. 웹 요청 시 외부 API 직접 호출 금지
웹 화면 요청 시점에 외부 API를 직접 호출하여 결과를 즉시 조합하는 방식은 기본 구조로 채택하지 않는다.

즉, 데이터 흐름은 아래 기준을 따른다.

1. 백엔드 내부 스케줄러 또는 수집 프로세스가 외부 API 호출
2. 필요한 데이터 저장 및 가공
3. 웹 요청은 저장된 데이터 조회

이 원칙을 유지하는 이유는 다음과 같다.

- 화면 응답 속도 안정화
- 외부 API 장애가 사용자 화면에 직접 전파되는 것 방지
- 수집 시점과 조회 시점 분리
- 화면 로직 단순화

### 2. 프론트엔드의 역할 제한
프론트엔드는 화면 렌더링과 사용자 상호작용에 집중한다.

즉, 아래와 같은 복잡한 계산은 가능하면 백엔드에서 처리한다.

- 순위 계산
- 기준값 비교
- 파생 지표 생성
- 화면 표 구성용 결과셋 조합

프론트엔드는 가공된 결과를 표시하는 역할에 가깝게 유지한다.

### 3. 초기 설계 우선순위
현재 우선순위는 아래와 같다.

1. 요구사항 기준의 화면/표 재현 구조 정리
2. 수집 및 저장 구조 정리
3. API 응답 구조 정리
4. 배포 및 운영 흐름 정리
5. 실제 소스 구현

즉, 현재 단계에서는 코드 작성보다 설계/구조/문서 기준 확정이 우선이다.



## 표별 구현 기준

아래 7개 표는 모두 최종 구현 범위에 포함한다.  
다만 표마다 데이터 취득 방식, 서버 가공 필요 여부, 저장 전략이 다르므로 이를 구분하여 설계한다.

### 1. 시장종합
- 목적: 지수, 등락, 거래대금 등 시장 전체 상태를 한눈에 보여주는 요약 영역
- 데이터 성격: 특정 시점의 대표 시장 상태
- 서버 가공 필요 여부: 낮음
- 저장 전략: `latest` 중심
- 수집 방식: 장중 주기 수집
- 조회 방식: 가장 최근 수집 시점 기준 단건 조회
- 비고: 화면 진입 시 가장 먼저 보여주는 상단 요약 영역 성격으로 본다

### 2. 투자자별 매매종합
- 목적: 개인, 외국인, 기관 등 주요 투자주체별 매매 흐름 요약 제공
- 데이터 성격: 특정 시점 기준 누적/요약 데이터
- 서버 가공 필요 여부: 낮음
- 저장 전략: `latest` 중심
- 수집 방식: 장중 주기 수집
- 조회 방식: 가장 최근 수집값 조회
- 비고: 시장종합과 함께 메인 요약 영역으로 사용한다

### 3. 장중 투자자별 매매 상위
- 목적: 투자주체별 매매 상위 종목을 시점별로 확인
- 데이터 성격: 시점별 랭킹 데이터
- 서버 가공 필요 여부: 보통
- 저장 전략: `snapshot`
- 수집 방식: 장중 주기 수집
- 조회 방식: 기준 시각별 목록 조회
- 비고: 특정 시각의 결과를 다시 볼 수 있어야 하므로 latest만으로는 부족하다

### 4. 프로그램매매 수상위
- 목적: 프로그램 매매 기준 상위 종목을 화면 요구사항에 맞게 제공
- 데이터 성격: 원본 응답만으로는 부족할 수 있는 가공형 결과셋
- 서버 가공 필요 여부: 높음
- 저장 전략: `snapshot`
- 수집 방식: 장중 주기 수집 후 서버 가공
- 조회 방식: 기준 시각별 결과 목록 조회
- 비고: 외부 API 원본을 그대로 노출하기보다, 서버에서 정렬/선별/계산한 최종 결과를 저장 및 조회하는 방향으로 본다

### 5. 종목별 프로그램매매추이
- 목적: 특정 종목의 프로그램 매매 흐름을 시간 축으로 확인
- 데이터 성격: 시계열 추이 데이터
- 서버 가공 필요 여부: 낮음 ~ 보통
- 저장 전략: `history`
- 수집 방식: 관심종목 기준 반복 수집
- 조회 방식: 종목 + 기간 기준 시계열 조회
- 비고: latest가 아니라 누적 이력 보관이 핵심이다

### 6. 종목별 공매도추이
- 목적: 특정 종목의 공매도 관련 추이를 시간 흐름에 따라 확인
- 데이터 성격: 시계열 추이 데이터
- 서버 가공 필요 여부: 낮음 ~ 보통
- 저장 전략: `history`
- 수집 방식: 관심종목 기준 반복 수집
- 조회 방식: 종목 + 기간 기준 시계열 조회
- 비고: 프로그램매매추이와 유사하게 history 중심으로 본다

### 7. 지수기여도상위
- 목적: 지수 상승/하락에 영향을 많이 준 종목을 순위 형태로 제공
- 데이터 성격: 서버 계산이 필요한 가공형 랭킹 데이터
- 서버 가공 필요 여부: 높음
- 저장 전략: `snapshot`
- 수집 방식: 장중 주기 수집 후 서버 가공
- 조회 방식: 기준 시각별 결과 목록 조회
- 비고: 단순 원본 조회가 아니라 서버에서 기여도 계산 또는 조합 로직을 거쳐 결과를 만든다

## 저장 전략 적용 기준

### 1. latest
현재 시점 기준 대표 상태만 필요하고, 과거 시점 복원이 중요하지 않은 표에 사용한다.

대상:
- 시장종합
- 투자자별 매매종합

### 2. snapshot
특정 시각의 결과 목록을 다시 확인할 필요가 있는 표에 사용한다.

대상:
- 장중 투자자별 매매 상위
- 프로그램매매 수상위
- 지수기여도상위

### 3. history
특정 종목의 시간 흐름을 조회해야 하는 시계열성 표에 사용한다.

대상:
- 종목별 프로그램매매추이
- 종목별 공매도추이

## 표별 구현 우선순위 해석

7개 표는 모두 구현 대상이다.  
다만 구현 난이도는 동일하지 않다.

- 상대적으로 직접 구성 가능한 표: 시장종합, 투자자별 매매종합, 장중 투자자별 매매 상위
- 관심종목 이력 관리가 중요한 표: 종목별 프로그램매매추이, 종목별 공매도추이
- 서버 계산/가공 비중이 큰 표: 프로그램매매 수상위, 지수기여도상위

즉, 일부 표를 제외하는 것이 아니라, 구현 시 서버 가공량과 저장 전략이 다르다는 점을 분명히 구분한다.
