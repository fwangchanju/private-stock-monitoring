# 화면별 구현 계획

## 1. 시장종합 — 설계 확정

- **API**: ka20001 (업종현재가요청)
- **수집 주기**: 장중 5분 주기 (collectMarketData) — 현행 유지
- **저장 전략**: latest (MarketOverview) + snapshot (MarketOverviewSnapshot)
- **수집 흐름**:
  1. 코스피(mrkt_tp=0, inds_cd=001), 코스닥(mrkt_tp=1, inds_cd=101) 각 1회 호출
  2. 응답에서 현재가, 등락률, 거래량, 거래대금, 상승/보합/하락 종목수 추출
  3. MarketOverview 엔티티 upsert + 시점별 snapshot 저장
- **현황**: Collector 구현 완료, 경로 수정 완료 (/api/dostk/sect)
- **남은 작업**: 실제 API 호출 테스트로 응답 래퍼 필드명(output?) 및 개별 필드명(pred_pre, mrkt_stat_cls_code 등) 검증

## 2. 투자자별매매종합 — 설계 확정

- **API**: ka10051 (업종별투자자순매수요청)
- **수집 주기**: 장중 5분 주기 (collectMarketData) — 현행 유지
- **저장 전략**: latest (InvestorTradingSummary) + snapshot
- **수집 흐름**:
  1. 코스피(mrkt_tp=0), 코스닥(mrkt_tp=1) 각 1회 호출 (base_dt=당일, amt_qty_tp=0(금액), stex_tp=1(KRX))
  2. 응답 inds_netprps[] 에서 종합지수 항목 필터링 (inds_cd=001 또는 101)
  3. 해당 항목의 ind_netprps(개인), frgnr_netprps(외국인), orgn_netprps(기관계) 순매수 금액 저장
- **제약**: 순매수 금액만 제공, 매수/매도 분리 없음 → 화면에서 순매수 금액만 표시
- **현황**: Collector 구현 완료, 경로·base_dt 수정 완료 (/api/dostk/sect)
- **남은 작업**: 실제 API 호출 테스트로 inds_cd 값(001/101) 및 응답 구조 검증

## 3. 장중투자자별매매상위 — 설계 확정

- **API**: ka10065 (장중투자자별매매상위요청)
- **수집 주기**: **18:30 이후 1회 배치** (현재 장중 5분 배치에 잘못 포함 → 분리 필요)
- **저장 전략**: snapshot

### 코드 수정 사항

**orgn_tp 코드 확정 (API 문서 기준)**:

| 코드 | 투자자 |
|---|---|
| 9000 | 외국인 |
| 9999 | 기관계 |
| 1000 | 금융투자 |
| 2000 | 보험 |
| 3000 | 투신 |
| 4000 | 은행 |
| 5000 | 기타금융 |
| 6000 | 연기금 |
| 7000 | 국가 |
| 7100 | 기타법인 |
| 9100 | 외국계 |

**개인 코드 없음** — ka10065는 기관/외국인 매매상위만 제공. 개인 투자자 조회 불가.

**현재 코드 문제점**:
- `InvestorType.PERSONAL` → "8000" 매핑: 잘못됨. 8000 코드 자체가 존재하지 않음
- `InvestorType.INSTITUTION` → "1000" 매핑: 금융투자만 조회됨. 기관 전체는 "9999"
- 장중 5분 스케줄러에 포함: 18:30 이전에는 데이터 미제공

**수정 방향**:
1. `InvestorType.PERSONAL` 수집 제거 (API 미지원)
2. `InvestorType.INSTITUTION` 코드를 "9999"(기관계)로 변경
3. `InvestorType.FOREIGNER` 코드 "9000" 유지
4. CollectionScheduler에서 장중 배치 → 18:30 이후 별도 배치로 분리
5. 수집 단위: (시장 2 × 투자자 2 × 매매방향 2) = 8회 API 호출

## 4. 프로그램순매수상위 — 설계 확정

- **API**: ka90003 (프로그램순매수상위50요청)
- **수집 주기**: 장중 5분 주기 (collectMarketData) — 현행 유지
- **저장 전략**: snapshot

### 화면 구성 (확정)

- **시장 필터**: 전체 / 코스피 / 코스닥 (탭 또는 세그먼트 전환)
- **금액·수량 전환**: 금액 / 수량 (토글 전환)
- **순매수·순매도 전환**: 순매수 / 순매도 (토글 전환)

### 수집 방식

- 코스피(P00101) + 코스닥(P10102) × 순매수/순매도 × 금액/수량 = **8회 API 호출**
- "전체"는 별도 API 호출 없이 코스피+코스닥 결과를 서버에서 합산·재정렬하여 제공
- snapshot 저장 시 시장구분(KOSPI/KOSDAQ) + 순매수/순매도 + 금액/수량 조합별로 저장

### 코드 수정 사항

1. 코스닥(P10102) 수집 추가
2. 금액(amt_qty_tp=1) + 수량(amt_qty_tp=2) 둘 다 수집
3. "전체" 합산 로직을 QueryService에서 처리
4. 필드명 이중정의 문제는 테스트로 확인 후 확정

## 5. 종목별 프로그램매매추이 — 설계 확정

- **API**: ka90008 (시간별) + ka90013 (일별)
- **저장 전략**: history (ProgramTradingHistory)
- **수집 단위**: 관심종목(WatchStock) 기준

### 화면 구성 (확정)

- **시간별·일별 전환**: 탭 또는 세그먼트 전환
- 시간별: 당일 시간대별 프로그램 매매 추이 차트
- 일별: 기간별 일 단위 프로그램 매매 추이 차트

### 수집 방식

- ka90008 (시간별): 장중 5분 주기 — 관심종목별 호출
- ka90013 (일별): 장외 16:00 배치 — 관심종목별 호출 (공매도 수집과 묶음)

### 코드 수정 사항

1. ka90008: 응답 배열에서 `tm` 필드 기준으로 최신 항목 판별 (순서 의존 제거)
2. ka90013: 수집 로직 신규 구현 + CollectionScheduler 장외 배치 추가
3. ProgramTradingHistory에 중복 저장 방어 추가 (existsByStockCodeAndSnapshotTime)
4. 시간별/일별 데이터 구분을 위해 엔티티에 타입 필드 추가 또는 테이블 분리 검토

# 현재 코드 구조상 문제점

## 1. IntradayInvestorRankingCollector 스케줄러 위치 + orgn_tp 매핑 오류 (높음)

- 장중 5분 배치에 포함 → 18:30 이전에는 데이터 미제공
- PERSONAL → "8000": 존재하지 않는 코드
- INSTITUTION → "1000": 금융투자만 조회됨, 기관 전체는 "9999"

## 2. 응답 래퍼 필드명 전체적으로 불확실 (높음)

MarketOverviewCollector의 `response.path("output")`, StockMasterCollector의 `response.path("list")` 등
실제 응답 래퍼가 맞는지 미확인. 틀려도 오류 없이 값이 0으로 조용히 저장됨.

## 3. ka90008 응답 배열 순서 불확실 (높음)

`outputList.get(0)` 으로 첫 번째 항목만 저장. 오름차순이면 항상 당일 최초 시각 데이터만 저장하게 됨.
→ `tm` 필드 기준 최신 항목 판별로 수정 필요.

## 4. ka90013 수집 코드 없음 (중간)

일별 프로그램매매추이 Collector 미구현, 스케줄러 배치 없음.

## 5. ProgramTradingCollector 코스닥 미수집 (중간) → 4번 설계에서 해결 예정

코스피(P00101)만 수집, 코스닥(P10102) 누락. → 코스피+코스닥+금액+수량 8회 호출로 변경 예정.

## 6. ProgramTradingHistory 중복 저장 방어 없음 (낮음)

동일 (stockCode, snapshotTime) 조합 중복 저장 방어 로직 없음.

## 7. IndexContributionRankingCollector 동작 불가 (별도)

추정값 TR ID + 경로로 장중 5분마다 잘못된 API 호출 중. 즉시 비활성화 필요.
